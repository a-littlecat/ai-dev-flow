# v0.8 Skill 瘦身与前沿模型净收益 RFC

> 状态：Draft for Review
>
> 关联任务：`PLAN-001`
>
> 日期：2026-07-19
> 适用范围：`ai-dev-flow` Skill 的默认使用路径、文档结构、模型调用策略和后续 v0.8 实施顺序

## 1. 决策摘要

`ai-dev-flow` 保留，但定位从“所有项目任务都进入完整工作流”收缩为“按风险启用的治理内核”。

本 RFC 建议：

1. 默认使用 `Lite`，只有跨会话、中高风险、真实环境、发布或多 agent 任务才升级到 `Tracked` / `Controlled`。
2. 当前执行会话所用模型负责规划、实现和自检；Skill 只保留状态、权限、证据、Git 边界和高风险停止条件。
3. 不因模型更强而自动增加 Reviewer、subagent、修复轮次或跨模型调用；只由确定性的风险审核闸门决定是否启动一个隔离的只读 Reviewer 上下文。
4. 先做零额度历史回放；只有回放通过后才允许制作不默认启用、可整体回退的最小原型，并使用当前执行会话所用模型完成最多 3 次真实任务对照；全部门槛通过后才允许全面收缩和迁移。
5. 原 PLAN-001 的 Review-Repair 研究只保留为本 RFC 的一个设计输入，不再维持独立 RFC 和 9 个后续任务。
6. 如果实测不能显著降低上下文、模型调用和用户打断，停止 v0.8 扩建，退回“精简 AGENTS + Git + 验证清单”。

一句话目标：让 Skill 只在能减少返工或风险时出现，不能为了证明流程存在而消耗额度。

### 1.1 模型术语约定

- **当前执行模型**：当前 Codex / agent 会话实际使用的模型；本计划不要求更换模型，也不要求新增 API。
- **当前模型真实任务对照**：由同一个当前执行模型分别按 `无 Skill + 精简 AGENTS`、`Lite`、`现行 Full` 执行冻结任务；最多 3 次。
- **额外模型供应商**：当前执行环境之外的其他模型 API、账号、CLI 或服务。本计划默认不接入、不调用，也不需要 API key。
- **独立 Reviewer**：与 Engineer 隔离的只读 agent / 上下文，可以使用同一平台和同一模型；“独立”指职责与上下文隔离，不代表接入额外模型供应商。

### 1.2 2026-07-19 续做修订：可获得的模型一致性证据

`V08-LEAN-EVAL-002` 的整体 Review 发现：当前 Codex harness 不向任务暴露精确 backend 模型版本或 opaque call ID。用户随后明确要求继续完成 v0.8，因此本 RFC 允许使用新 evaluation ID 做一次替代评估，但不得回写或合并旧 ledger。

新评估的同模型合同改为当前平台可真实证明的范围：

- 三次 main 均由同一父任务依次创建，`fork_turns=none`；
- 不传 `model`、`reasoning_effort` 或额外 provider override；
- 在运行前冻结三个 canonical agent path、执行顺序和 workflow bundle hash；
- 每次保存 spawn 请求形状、canonical agent path、前一 run 完成收据 hash、模型输出和验证证据；
- backend 精确版本明确记录为“平台未暴露”，因此结论只适用于同一父任务默认模型绑定，不能跨会话宣称精确模型版本相同。

这是新协议，不是追溯放宽 `V08-LEAN-EVAL-002`。旧评估继续保持 `Blocked` 历史结论。

## 2. 为什么 PLAN-001 不是瘦身方案

`PLAN-001` 解决的是固定两轮 Review-Repair 的停止决策问题。它提出软预算、progress/stall gate、能力降级和 Reviewer 路由，但没有降低：

- 默认读取多少文档；
- 小任务是否需要 TASK / TASK_BOARD；
- 一项任务会调用几个模型或 Reviewer；
- 用户会被多少次状态和授权问题打断；
- Skill 自身需要维护多少重复规则；
- v0.8 需要执行多少任务才能看到收益。

相反，当前规划新增了 1 份 RFC 和 9 个串行 Draft TASK。因此 `PLAN-001` 只能作为本 RFC 的 Loop 子模块输入，不能代表项目已经瘦身。

## 4. 净收益定义

```text
净收益 = 避免的返工 + 避免的误操作 + 保存的状态/证据
       - 文档读取成本 - 额外模型调用 - 用户打断 - 维护与迁移成本
```

Skill 只有在可观察结果为正时才应触发。模型能力、上下文窗口或厂商排名不能替代这个判断。

## 5. 三档目标架构

### 5.1 Lite：默认路径

适用条件必须全部满足：

- 单会话可完成；
- 预计修改不超过 3 个业务文件；
- 不涉及架构、依赖、数据库、发布、安全、权限、真实环境或不可逆动作；
- 无并行写入 agent；
- 有覆盖全部关键完成标准的确定性验证；“容易回滚”只能降低执行风险，不能替代验证。

行为：

- 不创建 TASK、TASK_BOARD 行或 UA 全量记录；
- 只记录目标、约束、完成标准、Git 状态、验证结果和最终 diff；
- 一个主模型完成实现与自检；
- 不自动调用独立 Reviewer、第二模型或 repair loop；
- 验证覆盖全部关键完成标准且通过、无阻断 finding 时立即结束；
- 任何关键结果仍需用户观察、体验判断或真实环境证据时，必须升级到 Tracked，并保留最小用户验收步骤。

### 5.2 Tracked：跨会话或中等风险

任一条件满足即升级：

- 跨会话；
- 修改超过 3 个业务文件或多个模块；
- B 级但回滚成本较高，或 C 级普通任务；
- 需要任务依赖、明确 base/diff 或独立 Review。

行为：

- 使用一份 Compact TASK，不默认复制 Full/Legacy 区块；
- TASK 是事实源，看板只投影 active task 的简短状态；
- 命中第 8.1 节风险审核闸门时，自动启动一个隔离的只读 Reviewer 上下文；未命中则不为流程完整性强制调用；
- 进入 Review-Repair 时基础预算 2，只有单调进展才逐次 +1，ceiling 3；
- 无进展、范围扩大或证据不足立即停止，不靠换模型重置预算。

### 5.3 Controlled：高风险完整治理

适用：

- D 级、架构、发布、安全、数据迁移；
- UA5 / UA6 / UA7；
- 真实环境、实机验收；
- 多写入 agent、Parallel Wave；
- 不可逆或外部状态变更。

行为：

- 保留 TASK、权限来源、证据、Review、UA、delivery 和隔离工作区；
- 交付或进入验收建议前必须通过第 8.1 节自动审核闸门；独立 Reviewer 仍需 authority 与 capability 同时满足；
- repair 基础预算为 2；满足第 8.2 节单调进展条件时自动增加且只增加 1 轮，ceiling 3；
- 用户只处理真实决策、实机证据和新授权，不处理纯粹的轮次耗尽。

## 8. 审核、修复与模型调用门禁

### 8.1 自动审核闸门

自动审核是 agent / harness 按规则执行的只读路由，不是后台调度服务。首版不需要数据库、队列、遥测系统、模型 Adapter 或看板 Loop 投影。

| 模式 | 自动审核规则 |
|---|---|
| Lite | 禁止自动 Reviewer；确定性验证和主模型自检后结束 |
| Tracked | 满足任一触发条件时自动启动一次初始 Review；否则不调用 Reviewer |
| Controlled | 进入验收建议、delivery、merge 或 release 建议前必须完成独立 Review |

Tracked 触发条件：

- 代码 diff 超过 3 个业务文件或跨多个模块；
- 修改公共 API、共享组件、构建/部署配置或核心执行路径；
- 确定性测试、lint、静态检查无法覆盖关键完成标准；
- 本任务曾修复 P0/P1，必须回到 Reviewer 复核；
- TASK、项目规则或用户已明确要求独立 Review。

以下情况不得为“流程完整”而自动审核：

- 纯文档、小范围、可逆且已有确定性验证的 Tracked 任务；
- 输入范围尚未冻结、diff 归属不清或验证结果尚未形成；
- 只是因为换用了更强模型、上下文更大或还有剩余额度；
- Reviewer 与 Engineer 不能形成职责或上下文隔离。

Reviewer 的最小输入只包含：冻结的目标与非目标、base/diff、修改文件、验证结果、项目硬约束和已有 finding。默认不得重新发送整个仓库。输出必须使用 P0～P3、稳定 finding ID、文件/行或证据位置，以及 `Passed / Needs Fix / Blocked` 结论。

“独立”指写权限和上下文隔离，不要求不同厂商：同一模型在新的只读上下文中可以担任 Reviewer；这不计为跨模型调用。Reviewer 可以写回结构化审查记录，但不得修改业务代码。

如果项目已经授权只读 Reviewer 且数据边界未变化，命中闸门后无需再次询问用户。涉及新外部服务、敏感数据边界或新增付费权限时仍须取得新授权。

Tracked 命中审核闸门后，Reviewer 的 authority 与 capability 都是硬门禁：

- authority 缺失或为 deny / unknown：进入 `Blocked`，等待项目政策或用户授权；
- capability 缺失：仅当升级 Controlled 能提供合法隔离 Reviewer 时才升级，否则保持 `Blocked`；
- 可以使用项目允许的人工或其他独立 Reviewer，但必须保持同一冻结 payload、只读边界和结果写回格式；
- 任何替代降级都必须由用户明确授权，且不得把“未执行 Review”记录为 `Passed`。

Controlled 缺少可用 Reviewer authority / capability 时同样必须标记 `Blocked` 或使用用户明确授权的独立 Reviewer，不能由 Engineer 自我批准。

同一冻结 payload 只允许一次初始 Review。进入 Review-Repair 后，每轮修复形成新 payload 并由同一 Reviewer 上下文复审；更换 Reviewer 或当前执行模型版本不得重置调用和轮次计数。

### 8.2 受控修复延长

Tracked 与 Controlled 的 repair 基础预算均为 2 轮。完成第 2 轮后，只有同时满足以下条件，才自动授予第 3 轮：

1. 目标、权限、依赖和 diff 边界保持冻结；
2. P0/P1 数量或严重度连续下降，没有新增同级或更高 finding；
3. RED / GREEN、测试、lint 或其他确定性证据相对上一轮改善；
4. 根因和剩余修复项已经明确，第 3 轮有具体闭合目标；
5. Reviewer 与 Repairer 仍具备所需 capability，且调用未突破项目成本和数据边界。

满足条件时由流程自动继续，不得仅因“已到两轮”要求用户查看。第 3 轮是绝对上限，不得再次延长。

出现任一情况立即停止并进入 `Blocked`、拆新 TASK 或请求真实决策：

- finding 数量/严重度无下降，或同一 finding 原样重复；
- 新增 P0/P1、测试回退、范围扩大、新依赖或架构变化；
- diff 归属不清、证据不足、根因重新变为未知；
- 需要新的权限、真实环境操作或不可逆外部动作；
- 第 3 轮复审后仍有 P0/P1。

修复代码获得第 3 轮，不等于自动重试发布、迁移、删除或其他外部副作用；不可逆动作仍服从原 authority gate。

### 8.3 模型调用与额度上限

每次新增 Reviewer、subagent、跨模型或额外 repair 调用前，必须同时满足：

1. 能说明该调用要降低的独特风险；
2. 该问题不能由测试、lint、静态检查或当前 diff 直接判定；
3. 输入范围已冻结，不发送整个仓库；
4. 预期输出与上一调用不重复；
5. 项目/用户允许对应成本和外部数据边界；
6. 调用后存在明确停止条件。

默认上限。Review 轮次包含一次初始 Review，以及每次有效 repair 后对新 payload 的复审：

| 模式 | 主模型上下文 | Reviewer 上下文 | 最大 Review 轮次 | 跨模型调用 | repair 上限 |
|---|---:|---:|---:|---:|---:|
| Lite | 1 | 0 | 0 | 0 | 0 |
| Tracked | 1 | 最多 1 | 未触发为 0；触发后最多 4 | 默认 0 | 基础 2；有 progress 时 3 |
| Controlled | 1 | 1 | 最多 4 | 默认 0，需明确理由和新授权 | 基础 2；有 progress 时 3 |

模型名称、上下文窗口和榜单结果不能提高这些上限。更换模型不能重置 attempt、repair 或成本计数。

### 9.2 阶段 B：有硬预算的小样本对照

只有阶段 A 通过后才执行：

- `LEAN-002` 只制作可整体删除、默认关闭、不迁移历史项目的最小原型；不得在真实收益通过前完成全量文档收缩、默认入口切换或兼容迁移；
- 冻结 1 个可逆、无敏感数据且所有完成标准都能确定性验证的 Lite 代表任务；任务、输入、验收 oracle 或模型版本变化即重新开始三次对照；
- 比较 `无 Skill + 精简 AGENTS`、`Lite`、`现行 Full`；
- 只使用当前执行会话所用模型，不接入其他模型 API、账号、CLI 或供应商；
- 当前模型真实任务对照总执行不超过 3 次；更换当前执行模型版本需新授权。

`V08-LEAN-EVAL-002` 的三次额度已经耗尽。用户本次“继续完成 v0.8”只额外授权 `V08-LEAN-EVAL-003` 一个替代周期，仍严格为 `no-skill -> lite -> full` 各一次 main；不允许继续追加第三个周期。

新替代周期开始前必须同时满足：

- 原型的路由、审核和 repair 决策来自原型文件内的单一机器可读 policy；
- 阶段 A 直接读取该 policy 计算 6 个 route 和 2 个 repair trace，不再使用与原型平行的硬编码规则；
- policy 明确覆盖 D 级、架构、数据迁移、安全、发布、真实环境和不可逆动作的 Controlled 路由；
- Tracked Review 覆盖公共 API、共享组件、构建/部署配置、核心路径和确定性验证缺口；
- Controlled 在验收建议、delivery、merge 和 release 前强制 Review；
- 新协议与 stage A 零额度结果先经独立 Review，无 P0/P1 后才允许新的三次 main。

三次执行必须使用同一代码基线、同一目标、同一完成标准、同一验证命令和同一输出评分表。执行顺序由 `LEAN-001` 在开始前固定；每次只运行一种模式，禁止边看结果边修改 Prompt 或 oracle。

### 9.3 可复现的计量协议与通过门槛

每个样本均写一行原始 ledger，不只写汇总结论：

| 指标 | 固定单位与采集方法 | 通过口径 |
|---|---|---|
| 工作流输入 | 实际加载的工作流/规则文本按 UTF-8 字节数和非空行数计；业务需求与代码 diff 单独记录，不混入 | Lite 相对 Full 两项均减少至少 50% |
| 模型调用 | 每次主模型、Reviewer、subagent 或重试请求各计 1；同一请求内部工具调用不另计 | Lite 相对 Full 减少至少 50%；Lite Reviewer=0 |
| 用户流程问题 | 需要用户回复才能继续的阻塞问题每个计 1；无需回复的状态通知不计 | Lite 相对 Full 减少至少 50% |
| 任务结果 | 冻结验证命令的 exit、通过数、输出文件 hash/结构和完成标准逐项布尔值 | Lite 不劣于 Full，三次均满足全部完成标准 |
| 审核质量 | 固定样本中历史 P0/P1、authority 和真实环境/delivery gate 的 expected/actual 标签 | 零 false negative；样本不足时只能写“固定样本未观察到漏检” |
| 自动审核 | 每次触发/跳过原因、payload hash、Reviewer 结论 | Tracked/Controlled 与 manifest 预期完全一致；Lite 调用数为 0 |
| repair 决策 | 每轮稳定 finding ID、严重度、验证结果、scope hash 和 extend/stop | 两个 trace 的实际决定与预期完全一致，最多第 3 轮 |
| 维护成本 | 活跃规范文件数、活跃规范非空行数、新增核心文件数 | 活跃核心 reference 不超过 12；新增核心文件不超过 2；活跃规范行数不得高于冻结基线 |
| 迁移成本 | 必需用户迁移步骤数、批量改写历史 TASK 数、新 required dependency 数、实施 TASK 数 | 用户迁移步骤不超过 3；历史 TASK 改写=0；required dependency=0；实施 TASK 不超过 3 |

`LEAN-001` 在任何原型前冻结基线值、统计脚本/命令和 ledger schema；`LEAN-002/003` 只能填充结果，不能追溯修改单位或阈值。

续做修订只允许新建 `V08-LEAN-EVAL-003` 协议与 ledger；不得修改 `V08-LEAN-EVAL-002` 的 manifest、原始 runs、summary 或 Reviewer 结论。两个 evaluation ID 的结果不得合并。

各能力分别作出有限结论：

- Lite：由三次同任务真实对照验证效率、质量和用户打断；
- Tracked / Controlled：由固定历史回放验证路由、审核和安全门禁，只能报告该 manifest 上的结果；
- 自动审核：只证明固定样本的触发/跳过和只读边界，不推断所有项目均无漏检；
- 第 3 轮 repair：只证明两个冻结 trace 的 extend/stop 决定，不宣称真实任务成功率提升；
- 维护/迁移：按实际 Git diff、活跃加载 manifest 和用户迁移步骤机械复算。

如果某模式没有冻结样本、oracle 不完整或样本数不足，结论必须写“证据不足 / 未验证”，不得写“不增加 P0/P1 漏检”“已证明净正收益”或据此建议全量发布。

Lite 相对现行 Full 必须同时满足：

- 工作流输入/必读说明减少至少 50%；
- 模型调用数和用户流程问题各减少至少 50%；
- 不增加 P0/P1 漏检、权限越界或状态误报；
- 任务结果和验证信号不劣于 Full。

如果 `无 Skill + 精简 AGENTS` 在 Lite 场景达到相同质量且成本更低，正式规定该类任务不启用 Skill。

### 9.4 全面实施门禁

`LEAN-002` 到阶段 B 结束前只能交付最小原型、固定 ledger 和对照证据。只有第 9.3 节所有适用门槛通过、无未验证核心能力且独立 Review 通过后，`LEAN-003` 才允许全面精简 Skill、切换默认入口、编写迁移说明并提交是否发布 v0.8 的最终决策。

任一门槛失败或证据不足时，删除/关闭最小原型，保留评估报告，停止全面收缩；不得以“已经投入实现成本”为理由继续。

## 11. 停止与退出条件

满足任一条件即停止继续扩建：

- 历史回放发现 Lite 会漏掉 P0/P1、authority 或真实环境门禁；
- 自动审核闸门漏掉历史 P0/P1，或在无需审核的低风险样本上产生不可接受的额外调用；
- 第 3 轮不能证明单调进展，或被用于重试不可逆外部动作；
- 需要自动调度器、数据库、遥测或计费系统才能判断收益；
- 为兼容每个模型新增核心分支；
- 对照测试未达到第 9.3 节门槛；
- 两个试点连续出现更多用户打断、更多范围漂移或更多状态误报；
- 维护新合同所需文档量超过被删除/退役的运行时文档量。

退出结果：保留 v0.7 只读 Contract、Git/验证/权限核心，把 Skill 收缩成精简 AGENTS 兼容包和少量按需指南，不再建设通用 Loop 平台。
