# v0.7 Workflow Contract RFC

> - 状态：Approved / 任务拆分已授权，实施未开始
> - 工作模式：`plan_task`
> - 目标版本：v0.7
> - 基线：`4a6c41781a028bf6c78c1283f16f5d120ee61ae1`
> - 日期：2026-07-11
>
> 本轮边界：只定义方案，不修改现有 Skill、模板、Prompt、脚本、版本号或分发副本。

## 1. 决策摘要

本 RFC 建议在 v0.7 引入一个深的 `Workflow Contract Module`，收口目前散落在 TASK、TASK_BOARD、状态机、Review、UA、delivery 和 Prompt 中的重复状态语义。

推荐方案是：

1. TASK 继续作为唯一细粒度事实源。
2. 每个 v0.7 TASK 使用唯一的 `## Workflow Contract` 受限 Markdown 键值块；8 个稳定状态字段始终存在，其余字段按真实事件出现。
3. 模块只暴露一个语义入口：`WorkflowContract.inspect(target) -> WorkflowReport`。
4. TASK_BOARD 降为由 TASK 生成或对照的只读投影，不再是第二真相源。
5. A/B 常规任务使用 Compact Core；C/D、Batch、Wave、真实环境信号按条件启用 Overlay。
6. v0.7 首批只做 Contract、Legacy Reader、只读 lint 和 drift 检查；不做状态写入器、通用插件运行时或双向同步。
7. GPT‑5.6、Structured Outputs、subagents 和工具调用只作为可选 Adapter / Harness Profile；确定性结论仍由本地规则产生。
8. 旧 TASK 永久可读，采用 `dual-read, single-write`；不批量改写，冲突时报告而不是猜测。

一句话判断：先把“什么状态是真的、为什么是真的”变成稳定接口，再谈自动化写入和跨平台同步。

## 2. 为什么现在需要这一步

### 2.1 已有规则足够丰富，但 Interface 太浅

当前项目已经覆盖状态机、验证、Review、UA、Batch、Wave、真实环境反馈和 harness 兼容。问题不再是“缺规则”，而是调用者要跨多个长文档理解同一件事，并在多个位置重复维护状态。

可直接观察到的重复包括：

- `skills/ai-dev-flow/references/TASK_TEMPLATE.md:272` 有“代码审查”。
- `skills/ai-dev-flow/references/TASK_TEMPLATE.md:287` 又有“Diff 审查”。
- `skills/ai-dev-flow/references/TASK_TEMPLATE.md:354` 有“合并状态”。
- `skills/ai-dev-flow/references/TASK_TEMPLATE.md:361` 又有“提交 / 合并”。
- `skills/ai-dev-flow/references/TASK_BOARD_TEMPLATE.md:3` 声明看板只保留索引和状态，但 `:54` 的完整视图约 30 列。

这些重复会产生三类实际问题：

- Locality 低：一次状态变化要理解和修改多个位置。
- 漂移难发现：两个字段都合法，但彼此矛盾。
- 模型能力浪费：GPT‑5.6 的推理被用于重新解释文档结构，而不是发现真正的风险、边界和缺失证据。

### 2.2 发布身份也需要先收口

仓库 `VERSION` 当前为 `0.5.2`，README 和 `CHANGELOG.md` 已包含 `0.6.0 Unreleased`。如果直接把 v0.7 行为混入当前文档，使用者无法判断哪些规则已发布、哪些只是设计。

因此，本 RFC 把“发布收口”列为实现 v0.7 前置任务，而不是在本轮顺手改版本。

## 3. 目标与非目标

### 3.1 目标

- 为 TASK 状态建立一个稳定、可读、可确定性校验的语义 Interface。
- 合并重复 Review 和 delivery 状态，保留 findings、命令、证据等正文细节。
- 明确 lifecycle、Review、UA、delivery、authority 互相正交。
- 让 A/B 常规任务显著变轻，让复杂度只在触发条件出现时加载。
- 保守读取 v0.2–v0.6 风格 TASK，不强制迁移。
- 让不同 harness、模型和输出格式共享同一 Contract 语义。
- 为后续 fixture、eval、CI 和分发校验提供确定性基座。

### 3.2 非目标

- 不把项目改造成数据库、工作流引擎或通用编排平台。
- 不在 v0.7 首批实现自动状态流转、自动 merge、自动 push、自动 release。
- 不做 TASK 与 GitHub Issue、Notion 或其他外部系统的双向同步。
- 不把 GPT‑5.6 或任一模型名写进核心 schema。
- 不要求每个项目使用 subagent、Worktree、Batch 或 Wave。
- 不批量重写已有 TASK。
- 不把所有现有模板字段塞入新 Contract 块。

## 4. 硬约束

以下约束不是实现偏好，而是方案验收条件：

- Markdown-first：TASK 仍可被普通文本编辑器和普通 agent 直接读写。
- Agent-agnostic：核心语义不依赖 Codex、Claude、Gemini 或具体 API。
- Offline-capable：无网络、无模型 API 时，读取和确定性 lint 仍可运行。
- Read-only first：v0.7 首个工具只读，不改 TASK、TASK_BOARD 或 Git。
- Backward compatible：旧 TASK 可读；新严格规则只对显式声明 v0.7 的文件生效。
- Single source：Normalized View 只存在内存中，不额外落盘成 JSON/YAML 副本。
- Evidence before assertion：结构合法不等于 Review、UA 或 Closed 已完成。
- Authority separated：能力、权限、风险建议不可折叠成一个布尔值。
- No hidden widening：Overlay 只能补默认或收紧约束，不能放宽核心安全规则。
- No speculative seams：只有出现至少两个真实 Adapter 时才抽象公共 Seam。

## 5. 三种候选方案比较

| 方案 | Interface | 优点 | 主要代价 | 结论 |
|---|---|---|---|---|
| 最小单入口 | 仅 `inspect(target)`，沿用稳定 Markdown 标题 | 核心最小、Locality 强、实现简单 | 标题本身成为协议，A/B 模板仍可能偏长 | 采用“单一语义入口” |
| Compact Core + Overlay | 一个紧凑状态块，复杂路径按需升级 | 常规任务最轻，复杂规则仍保留 | 需要控制 Overlay 数量和重复真相 | 采用“按需 Overlay” |
| 扩展优先 | 状态块 + Project Overlay + Harness Profile + Adapter preview | 跨 harness 和未来扩展更清晰 | 容易过早形成插件框架 | 只采用有限扩展边界 |

最终选择是三者混合：

- 对调用者：一个 `inspect()` 语义入口。
- 对文件：一个 `## Workflow Contract` 状态块。
- 对复杂任务：有限的、按需加载的 Overlay。
- 对未来扩展：只有 optional/required 两种声明，不建立通用注册器。

## 6. Workflow Contract Module

### 6.1 Module 边界

```text
TASK Markdown ──> Markdown Reader ─┐
Legacy TASK ────> Legacy Reader ───┼─> Workflow Contract Module
Project Overlay ─> Overlay Reader ─┘            +─> WorkflowReport
                                                 +─> Human Output Adapter
                                                 +─> JSON Output Adapter
                                                 +─> TASK_BOARD Projection Adapter
```

核心 Module 负责：

- 读取并归一化 Contract。
- 保存每个字段的 provenance（来源文件、标题、行号）。
- 校验枚举、状态机和跨维度不变量。
- 对照 TASK_BOARD 投影。
- 输出不可变 `WorkflowReport`。

项目根目录扫描时，Overlay Reader 只按约定查找 `docs/PROJECT_OVERLAY.md`；扫描单个 TASK 时，先定位最近的 Git 项目根目录，再使用同一规则。路径不存在时不报错，也不搜索任意文件。Overlay 规则解析在 `CONTRACT-007` 才启用；此前如发现该文件，只输出 `W_PROJECT_OVERLAY_UNEVALUATED`，且报告不得声称已检查项目专用规则。Harness Profile 属于调用 `inspect()` 之前的 harness/orchestrator 上下文，不进入 Contract Module，也不改变 diagnostics。

核心 Module 不负责：

- 修改 TASK 或 TASK_BOARD。
- 决定是否 merge、push 或 release。
- 调用外部模型或平台。
- 自动解决含糊的 legacy 冲突。

### 6.2 唯一语义入口

```text
WorkflowContract.inspect(target) -> WorkflowReport
```

`target` 只接受：

- 单个 TASK 文件；或
- 项目根目录。

概念返回值：

```text
WorkflowReport
├─ contracts[]       # 归一化的任务状态快照
├─ diagnostics[]     # 诊断码、级别、说明和来源位置
├─ projections[]     # 可用时的 TASK_BOARD 对照结果
└─ summary            # TASK 数、错误数、违规数、警告数、漂移数
```

CLI、Human/JSON 输出和 GPT‑5.6 解释都只是 Adapter，不增加第二个核心入口。

## 7. Canonical Markdown Interface

### 7.1 为什么选受限 Markdown 键值块

本 RFC 推荐受限 Markdown 键值块，不推荐 YAML frontmatter、独立 JSON sidecar 或自由文本标题推断：

- 比 YAML 更少依赖和解析歧义；第一版可使用 Python 标准库实现。
- 比独立 JSON 更符合 Markdown-first，也避免双真相源。
- 比自由标题更稳定，字段名可版本化，正文仍保持中文和可读性。
- GPT‑5.6 可在内存中生成结构化 JSON，再由 Adapter 转成同一 Markdown 形状；落盘格式不绑定模型。

### 7.2 v0.7 状态块

```markdown
## Workflow Contract

- `schema_version`: `adf/v0.7.0`
- `task_id`: `TASK-123`
- `task_type`: `code`
- `task_class`: `B`
- `lifecycle`: `Ready`
- `review_status`: `Pending`
- `ua_level`: `UA3`
- `ua_status`: `Pending`
```

约束：

- 字段名和枚举使用稳定英文 token；正文说明继续使用中文。
- 状态块只保存短、可枚举、需要跨工具共享的事实。
- 目标、范围、完成标准、验证计划、findings、UA 操作和证据继续保留为 Markdown 正文。
- 不允许在同一 TASK 出现第二个 Workflow Contract 块。
- `schema_version` 是 Contract 语义版本，不等于 Skill 的发布 `VERSION`。
- 每个字段独占一行，语法固定为 ``- `key`: `value` ``；Reader 对字段顺序不敏感，但 Writer 使用上例顺序。
- 字段值不得包含换行或反引号；多值使用分号 `;` 分隔，`none` 必须单独出现。
- 未知字段在 `0.7.x` 中报错；只有第 9.2 节声明的扩展字段可以额外出现。
- key 和枚举值大小写敏感；键值行不得有额外前导/尾随空白。
- 同一 key 重复出现时输出 `E_PARSE`，不使用首值或末值。

已知字段集合固定为：

- 始终字段：`schema_version`、`task_id`、`task_type`、`task_class`、`lifecycle`、`review_status`、`ua_level`、`ua_status`。
- 条件字段：`ua_evidence`、`acceptance_authority`、`close_authority`、`commit_status`、`merge_status`、`merge_authority`、`overlays`。
- 扩展声明：`extensions_optional`、`extensions_required`。

`schema_version` 在本版只能是 `adf/v0.7.0`。`task_id` 必须是项目内非空且唯一的任务标识，并与文件名和 TASK_BOARD 投影一致；具体前缀由项目约定，不在核心强制为 `TASK-`。

### 7.3 Canonical 枚举

第一版只接受以下 token；中文只作为界面说明，不进入机器值：

| 字段 | v0.7.0 值域 |
|---|---|
| `task_type` | `document / plan / code / review / repair / test` |
| `task_class` | `A / B / C / D` |
| `lifecycle` | `Draft / Ready / In Progress / Blocked / Review / Needs Fix / Accepted / Closed / Deferred / Cancelled` |
| `review_status` | `Pending / In Review / Passed / Needs Fix / Do Not Merge` |
| `ua_level` | `UA0`–`UA7 / TBD` |
| `ua_status` | `Not Required / Pending / Passed / Failed / Deferred / TBD` |
| `acceptance_authority` | `None / User Confirmed / Designated Acceptor Confirmed` |
| `close_authority` | `None / User Authorized / Rule Authorized / Denied` |
| `commit_status` | `Not Applicable / Uncommitted / Committed` |
| `merge_status` | `Not Applicable / Unmerged / Merged / Deferred` |
| `merge_authority` | `None / User Authorized / Denied` |
| `overlays` | `none` 或 Registry 中已知 Overlay 的引用列表 |

`task_type` 只描述交付物性质，`task_class` 描述复杂度，两者不能互相推断。`mode` 是当前调用的执行上下文，不是持久任务状态，因此不进入 Contract；它继续复用 `GLOSSARY.zh-CN.md` 中的 mode id，由 Prompt/harness 在调用时提供。后续如需增加 Contract 值，必须先在语义规范、fixture 和所有 Reader 中同步。

`ua_evidence` 不是枚举：它保存一个或多个用分号分隔的仓库相对路径、Markdown 锚点或项目约定证据 id。Validator 可以检查引用是否存在以及是否来自期望区块，但不能证明记录者就是用户；当 authority 真实性无法从当前输入证明时，必须输出 `W_AUTHORITY_UNVERIFIABLE`，不得把结构存在写成“已确认”。

### 7.4 条件必填内容

| 条件 | 必须存在的语义内容 |
|---|---|
| 所有 v0.7 TASK | 状态块中的 8 个核心字段 |
| `Ready` 起 | 目标、非目标、允许修改、禁止修改、完成标准、验证方式 |
| 代码任务进入 `In Progress` 前 | Base commit；C/D 还需要隔离与回滚信息 |
| `Review` 起 | Diff 范围、修改文件、验证证据、唯一 Review 状态与 findings 引用 |
| `ua_status=Passed/Failed/Deferred` | `ua_evidence`：可定位的 Markdown 锚点、文件或证据引用 |
| `ua_status=Passed` 或 `Accepted` | `acceptance_authority`；UA7 只能是 `User Confirmed` |
| `ua_level=UA7` 且结果不是 `Pending/TBD` | `acceptance_authority=User Confirmed`；同意、拒绝和延期都不能由 agent 或项目规则代替 |
| `Needs Fix` | 可定位的 Review finding，或 Acceptance Feedback Gate 的分类、当前 TASK 范围结论和反馈证据 |
| `Accepted` | Review 门禁满足；当前 UA 等级要求的动作和证据完成；用户或项目明确指定的验收人确认可接受 |
| `Closed` | 已 Accepted；`close_authority` 明确为用户或项目规则授权 |
| Git 项目进入 `Review` | `commit_status`；若存在分支合并路径或已发生 merge，再要求 `merge_status` |
| `merge_status=Merged` | `merge_authority=User Authorized`、合并目标和事实证据 |
| 触发复杂路径 | `overlays`；未触发时省略并归一化为 `none` |
| `real_env_signal` Overlay | Feedback Gate 分类、是否属于当前 TASK、RED、GREEN、SIGNAL；需要时补 HITL 操作 |

条件字段仍写在同一个 Contract 块中。例如任务进入验收/交付阶段后，可在 8 个核心字段后追加：

```markdown
- `ua_evidence`: `#用户验收反馈`
- `acceptance_authority`: `User Confirmed`
- `commit_status`: `Committed`
- `merge_status`: `Unmerged`
- `merge_authority`: `None`
- `close_authority`: `None`
```

不得为了“字段齐全”提前写一排 `none` / `Not Applicable`。省略 `overlays` 可安全归一化为 `none`，省略 authority 可保守归一化为 `None`；省略 delivery 字段只能在内存 View 中记为 `Not Recorded`，不能猜成 `Not Applicable`，且 `Not Recorded` 不是合法落盘 token。所有默认值都要在 `WorkflowReport.provenance` 中标记为 default，不能伪装成文件原值。

### 7.5 推荐正文骨架

```markdown
# TASK-123：任务标题

## Workflow Contract

<!-- 固定状态块 -->

## 目标与边界

- 目标：
- 非目标：
- 允许修改：
- 禁止修改：

## 完成标准与验证

- 完成标准：
- 验证命令或检查：

## Outcome

- Base / Diff：
- 修改文件：
- 验证证据：
- Review findings：
- UA 动作与结果：
```

A/B TASK 默认到此为止；未触发的复杂路径不加载额外段落。

## 8. 状态轴和不变量

### 8.1 正交状态轴

当相应维度适用或事实发生时，必须分别记录：

- `lifecycle`：任务处于流程的哪个阶段。
- `review_status`：独立审查结论。
- `ua_level` / `ua_status` / `ua_evidence`：需要什么用户动作、动作是否完成、证据在哪里。
- `acceptance_authority`：谁确认验收结果可接受；UA7 必须由用户确认。
- `commit_status` / `merge_status`：Git 交付事实。
- `close_authority` / `merge_authority`：谁授权了动作。

不得进行以下推断：

- 自动化 GREEN ⇒ Review Passed。
- Review Passed ⇒ UA Passed。
- UA0 ⇒ 自动 Closed。
- Committed 或 Merged ⇒ Accepted。
- Merged ⇒ Closed。

如果现实记录为“已合并、未验收”，Contract 必须如实保留两个事实，同时报告顺序违规；不能把状态改写得更好看。

### 8.2 lifecycle

沿用现有 `STATUS_MACHINE.md`，不在 v0.7 新造状态：

```text
Draft -> Ready
Draft -> Deferred / Cancelled
Ready -> In Progress / Blocked
In Progress -> Review / Blocked / Deferred
Review -> Needs Fix / Accepted / Blocked
Needs Fix -> In Progress / Review
Accepted -> Closed
Blocked -> Ready / Deferred
Deferred -> Ready / Cancelled
```

以上与当前 `skills/ai-dev-flow/references/STATUS_MACHINE.md` 的允许流转保持一致；未来若状态机改变，Contract fixture 必须在同一变更中更新。

### 8.3 跨轴不变量

1. TASK 是事实源，TASK_BOARD 只能投影。
2. `review_status=Needs Fix/Do Not Merge` 不得与 `Accepted/Closed` 并存。
3. `Accepted` 必须满足适用于当前任务的 Review 和 UA 门禁，并由用户或项目明确指定的验收人确认。
4. `Closed` 必须来自 `Accepted`，并具有用户或项目规则授权。
5. UA0 只能提供关闭建议，不能自行产生关闭授权。
6. delivery 记录事实，不反推 lifecycle。
7. UA4–UA7 验收失败后，必须先通过 Acceptance Feedback Gate 分类为“原任务未完成”或“本轮回归”，并确认属于当前 TASK，才可建议进入当前任务的修复路径。
8. 验收失败进入修复前还必须形成可复现的 RED、预期 GREEN 和可信 SIGNAL；新需求、范围扩大、环境/配置问题和证据不足不得带入当前 TASK 盲修。
9. 已 Accepted 或 Closed 后收到失败反馈，默认新建回归 TASK 或等待用户明确授权，不把原任务直接跳回 Review/Needs Fix。
10. UA7 的 Passed、Failed 或 Deferred 都必须有 `User Confirmed`；agent 不能替用户同意、拒绝或延期。
11. `merge_status=Merged` 只接受 `merge_authority=User Authorized`；项目规则不能替代本次用户 merge 确认。
12. 结构 lint 通过不等于 Review、验收或交付已经完成。
13. Legacy 字段矛盾时必须报告冲突，不能使用“最后出现的值”。
14. 不认识 required extension 时任务标记为 Blocked；不认识 optional extension 时警告并保留原值。

## 9. Overlay：只在复杂度真实出现时升级

```text
Core TASK
├─ A/B：无 Overlay
├─ C/D：isolation Overlay
├─ Batch：batch reference Overlay
├─ Wave：wave reference Overlay
└─ 验收失败/实机问题：real_env_signal Overlay
```

### 9.1 Overlay Registry

| Overlay | 触发条件 | TASK 保存什么 | 共享事实放哪里 |
|---|---|---|---|
| `isolation` | C/D 或高风险代码任务 | 分支/Worktree、依赖、回滚、独立审查要求 | TASK |
| `batch` | 多个可顺序执行的 A/B 任务 | BATCH 文件引用 | BATCH 文件保存顺序和 per-task diff |
| `wave` | 经用户确认的并行工作 | WAVE 文件引用 | WAVE 文件保存锁、依赖、Worktree 和 Review Hub |
| `real_env_signal` | 用户实机失败、跨环境问题、UA4–UA7 失败 | signal 文件或内联段落引用 | signal 记录保存 RED/GREEN/SIGNAL/HITL |

`real_env_signal` Overlay 在 `CONTRACT-007` 至少定义以下稳定字段：

```markdown
- `feedback_class`: `regression`
- `task_scope`: `current`
- `red_ref`: `#RED`
- `green_ref`: `#GREEN`
- `signal_ref`: `#用户回传证据`
```

`feedback_class` 值域是 `original_incomplete / regression / scope_expansion / environment / insufficient_evidence`；`task_scope` 值域是 `current / outside / unknown`。只有 `feedback_class=original_incomplete/regression` 且 `task_scope=current` 时，才可能建议把当前任务转入 repair 路径；其余分类分别路由到新 TASK、Blocked、bug_diagnosis 或补充证据。该路由仍受 Accepted/Closed 后回归规则和用户授权约束。

Overlay 规则：

- Batch 与 Wave 是组织方式，不是 lifecycle。
- Batch 只接受 A/B，并且是顺序执行。
- D 级任务不得与其他代码任务并行；允许与只读分析、文档整理或审查任务组成 Wave，但必须保留高风险隔离和用户确认。
- 同一执行阶段 Batch 与 Wave 互斥。
- TASK 不复制 BATCH/WAVE 中的共享锁和调度细节。
- Project Overlay 只能补默认或收紧核心规则，不能放宽。
- 第一阶段不创建通用 Overlay 注册器；只实现文档中明确列出的有限集合。

### 9.2 有限扩展声明

如果未来出现项目专用扩展，只允许在同一 Contract 块追加两个可选字段：

```markdown
- `extensions_optional`: `project.real-env@1`
- `extensions_required`: `none`
```

语义如下：

- `extensions_optional`：不认识时警告，核心流程可继续。
- `extensions_required`：不认识时 Blocked。

命名格式：`<owner>.<name>@<interface-version>`，例如 `project.real-env@1`。

扩展不得重定义 lifecycle、Review、UA、acceptance、delivery 或权限语义。删除扩展后，核心 TASK 仍必须可理解。

## 10. Can / May / Should 分离

### 10.1 Harness Profile 只描述 Can

Harness Profile 可声明：

```markdown
- `can`: `git; worktree; readonly_subagents; structured_output`
- `cannot`: `real_environment_control`
- `degrade_to`: `serial_sessions`
```

它只能回答“当前 harness 能做什么”，不能授予权限。

Harness Profile 不由 `WorkflowContract.inspect(target)` 自动发现，也不参与 Contract pass/fail。它由调用方在执行前评估；v0.7 首批不新增 Profile 文件或 Profile validator，避免把环境能力混入持久任务事实。

### 10.2 权限来源只回答 May

push、release、外部写入、多写入 agent 等权限必须来自：

- 用户当前明确指令；或
- 项目已建立且适用于当前范围的规则。

merge 是更严格的例外：依据当前 `GIT_WORKFLOW.md` 和 UA7 规则，实际 merge 必须有本次用户确认，项目规则不能替代。

### 10.3 风险判断只回答 Should

即使 Can 和 May 都满足，agent 仍要依据任务等级、工作区状态、证据和回滚能力决定是否应该执行。三者冲突时采取更保守路径并报告原因。

## 11. Legacy 兼容与迁移

### 11.1 Reader 选择

```text
有 schema_version=adf/v0.7.0 -> V07MarkdownAdapter -> 严格校验
无 schema_version             -> LegacyMarkdownAdapter -> 保守归一化 + warning
```

Legacy Reader 要隐藏：

- 中英文标题和旧字段别名。
- “代码审查 / Diff 审查”的保守归一化。
- 两组合并字段的保守归一化。
- 旧 UA 表达向 `ua_level` / `ua_status` 的映射。

若多个旧字段一致，可以归一化并标记 provenance；若矛盾，输出 `E_LEGACY_CONFLICT`，不得猜测。

### 11.2 迁移原则

- `dual-read, single-write`：读 legacy 和 v0.7；新写只生成 v0.7。
- 旧 TASK 只在下一次真实修改时选择性升级，不批量重写。
- 首次升级先生成 preview；用户或任务执行者确认状态一致后再局部修改。
- 不自动重排整份 Markdown。
- `0.7.x` 只增加可选字段或澄清规则；删除、改名或改变核心语义需升级到下一 schema minor。
- Skill `VERSION` 与 Contract `schema_version` 分开管理。

## 12. 只读 workflow_lint

### 12.1 概念调用

```powershell
# 单个 TASK
python scripts/workflow_lint.py docs/tasks/TASK-042.md

# 项目扫描，并对照 TASK_BOARD
python scripts/workflow_lint.py .

# 同一 Interface 的 JSON 输出 Adapter
python scripts/workflow_lint.py . --format json
```

工具内部只调用：

```text
WorkflowContract.inspect(target)
```

### 12.2 检查顺序

```text
定位 TASK
-> 选择 Markdown Reader
-> 归一化并记录 provenance
-> 校验必填字段和枚举
-> 校验 lifecycle
-> 校验跨轴不变量
-> Git 可用时验证可证明的历史流转
-> 对照 TASK_BOARD
-> 输出只读 WorkflowReport
```

### 12.3 初始诊断码

| 代码 | 含义 |
|---|---|
| `E_PARSE` | Contract 结构无法解析或字段重复 |
| `E_UNKNOWN_VALUE` | 枚举值未知 |
| `E_TASK_ID_CONFLICT` | 文件名、Contract、看板编号冲突 |
| `E_LEGACY_CONFLICT` | 多个 legacy 状态字段互相矛盾 |
| `V_STATE_GUARD` | 当前 lifecycle 缺少条件必填内容或前置门禁 |
| `V_ILLEGAL_TRANSITION` | Git 历史可证明 lifecycle 非法跳转 |
| `V_REVIEW_GUARD` | 未满足 Review 门禁却进入 Accepted/Closed |
| `V_UA_GUARD` | 未满足当前 UA 动作、证据或确认来源却进入 Accepted/Closed |
| `V_CLOSE_AUTHORITY` | Closed 没有授权来源 |
| `V_ACCEPTANCE_SCOPE` | 验收反馈未分类、不是当前 TASK 范围，或已 Accepted/Closed 却复用原任务 |
| `V_SIGNAL_GATE` | 验收失败后无 RED/GREEN/SIGNAL 就进入修复 |
| `V_DELIVERY_ORDER` | delivery 发生在所需门禁之前 |
| `V_DELIVERY_AUTHORITY` | merge 事实缺少适用的授权来源 |
| `V_BOARD_DRIFT` | TASK_BOARD 与 TASK 不一致 |
| `V_OVERLAY_WEAKENS_CORE` | Overlay 放宽了核心规则 |
| `W_LEGACY_INFERRED` | 值来自 legacy 保守推断 |
| `W_TRANSITION_UNVERIFIABLE` | 缺少 Git 历史，无法证明上次状态 |
| `W_AUTHORITY_UNVERIFIABLE` | authority 字段存在，但当前输入不能证明确认主体 |
| `W_BOARD_MISSING` | TASK 没有看板投影 |
| `W_PROJECT_OVERLAY_UNEVALUATED` | 发现项目 Overlay，但当前阶段未启用其确定性解析 |

建议退出码：

- `0`：无 error / violation；warning 不阻塞。
- `1`：存在工作流违规。
- `2`：解析失败或命令调用错误。

Validator 必须明确写出：“lint 通过只代表 Contract 结构和可确定规则通过，不代表 Review、UA、merge 或 release 已完成。”

`V_OVERLAY_WEAKENS_CORE` 仅在 `CONTRACT-007` 提供确定性 Overlay Reader 后启用。Harness Profile 的 `Can`/`May` 混淆属于调用方行为 eval，不是 `workflow_lint` diagnostic。

## 13. GPT‑5.6 与高级 agent 能力

官方模型页显示 GPT‑5.6 支持 Structured Outputs、function calling，以及 Responses API 下的多种工具能力。项目可以利用这些能力，但必须放在核心 Contract 之外：[GPT‑5.6 Sol 模型页](https://developers.openai.com/api/docs/models/gpt-5.6-sol)。

### 13.1 推荐用法

- Contract drafting：让模型先输出符合 JSON Schema 的内存对象，再由确定性 Adapter 渲染为 Markdown patch。
- Evidence synthesis：把长日志、diff、测试输出归纳成候选 evidence refs，但引用必须可追溯。
- Cross-document review：并行只读分析 TASK、状态机、Prompt 和分发副本，由主 agent 归并结论。
- Ambiguity detection：发现 legacy 字段冲突、缺失授权、状态轴折叠和不可证明的断言。
- Bounded repair proposal：根据确定性 diagnostics 生成修复建议；不直接修改状态。
- Eval generation：从 Contract 不变量生成正例、反例和 adversarial fixture。

### 13.2 明确禁止

GPT‑5.6 Adapter 不得：

- 决定 lint pass/fail 或修改诊断严重度。
- 把模型推断写成用户授权。
- 把 review、UA、delivery 自动折叠为 lifecycle。
- 在证据不足时猜测 legacy 冲突。
- 直接写 TASK、TASK_BOARD、Git 或外部系统。
- 成为核心流程的必需依赖。

### 13.3 能力 Profile，不绑定模型名

核心只识别诸如 `structured_output`、`readonly_subagents`、`tool_calling`、`large_context_review` 等能力；具体模型与 reasoning 配置属于 harness Adapter。这样未来切换模型时无需升级 Contract schema。

## 14. Eval 与回归矩阵

v0.7 不应只靠文档 review。实现阶段至少建立以下 golden fixtures：

| Fixture | 期望结果 |
|---|---|
| 合法 A 级文档任务 | 通过，无 Overlay |
| Ready 但无验证方式 | `V_STATE_GUARD` |
| Review Passed、UA Pending | 保持 Review 或 Accepted 门禁失败，不得 Closed |
| UA Passed 但无 `ua_evidence` / `acceptance_authority` | `V_UA_GUARD` |
| UA7 Passed/Failed/Deferred 但不是 `User Confirmed` | `V_UA_GUARD` |
| 已 merge、未验收 | 如实保留 Merged，报告 delivery 顺序违规 |
| Merged 但 `merge_authority=None/Denied` | `V_DELIVERY_AUTHORITY` |
| UA0 Passed、无关闭授权 | 不得 Closed |
| Review Needs Fix、lifecycle Accepted | `V_REVIEW_GUARD` |
| UA5 失败后无 RED/GREEN/SIGNAL 进入修复 | `V_SIGNAL_GATE` |
| UA5 失败被分类为新需求仍进入当前任务修复 | `V_ACCEPTANCE_SCOPE` |
| Accepted 后收到失败反馈并把原任务改回 Needs Fix | `V_ACCEPTANCE_SCOPE` |
| TASK 与 TASK_BOARD 状态不同 | `V_BOARD_DRIFT` |
| legacy 两个 merge 字段冲突 | `E_LEGACY_CONFLICT` |
| 未知 optional extension | warning，继续读取 |
| 未知 required extension | Blocked / violation |
| Overlay 放宽核心禁止项 | `V_OVERLAY_WEAKENS_CORE` |
| Harness 有能力但无权限 | 不得执行外部写入 |
| Git 不可用 | 当前组合仍校验，历史流转给 warning |
| JSON 与 Human Adapter | diagnostics 语义完全一致 |
| GPT‑5.6 解释 Adapter | 可改变表述，不得改变诊断集合 |

还应增加三类 agent eval：

- Reviewer 不顺手修复。
- 自动化全绿不宣称用户验收通过。
- 多 agent 并行时，写入锁和最终状态只由单一 owner 收口。

## 17. 验收标准

RFC 获批后，实现工作必须满足：

- 新 A/B TASK 只需读取 Compact Core 和少量正文即可执行。
- 至少 3 个真实 A/B TASK 对比证明新模板填写项和重复状态少于旧模板；不能只用人为构造的最短示例。
- lifecycle、Review、UA、delivery、authority 有唯一规范字段。
- TASK_TEMPLATE 不再保存两套 Review 状态或两套 merge 状态。
- Compact Template 启用时，SKILL/PROMPTS 已停止强制写回被移除的旧状态段落。
- TASK_BOARD 明确为投影；漂移可被只读工具定位。
- legacy TASK 无需迁移即可读取；冲突不会被静默吞掉。
- `workflow_lint` 对同一输入的 Human/JSON 结果语义一致。
- 所有诊断包含来源位置和可执行修复建议。
- Validator 不写文件、不改 Git、不调用网络。
- 无 GPT‑5.6、无 subagent 时核心功能完整。
- 使用 GPT‑5.6 时只能提高结构化 drafting、跨文档审查和解释质量，不能改变确定性门禁。
- 文档、Skill、Prompt、脚本、fixture 和 VERSION 之间有自动或半自动一致性检查。

## 18. 风险与控制

| 风险 | 影响 | 控制 |
|---|---|---|
| Interface creep | 新块再次变成完整大模板 | 核心只存稳定状态；正文和 Overlay 按需加载 |
| Duplicate truth | TASK、正文、看板同时维护状态 | v0.7 状态只在 Contract；看板只投影；正文只留证据 |
| Parser fragility | Markdown 微小变化导致误判 | 受限键值格式、golden fixtures、provenance |
| Legacy ambiguity | 旧字段冲突被错误归一化 | `E_LEGACY_CONFLICT`，禁止猜测 |
| Overlay explosion | 形成隐性插件平台 | 固定有限集合；第二真实需求前不抽象新 Seam |
| Authority leak | Harness 能力变成操作许可 | Can/May/Should 分离，固定诊断 |
| Validator overclaim | lint 通过被写成已验收 | 输出强制免责声明，eval 覆盖 |
| Model coupling | schema 跟模型升级一起漂移 | 核心只识别能力；模型名留在 Adapter |
| Release confusion | v0.6/v0.7 行为混杂 | `REL-001` 前置；schema 和包版本分离 |
| Migration noise | 大量旧文件 diff 难审查 | 触碰时迁移、preview、局部 patch、不批量重写 |

## 19. 回滚与停线条件

实现阶段出现以下任一情况，应停止继续迁移并回到 RFC：

- 无法在不引入重量级 parser 的前提下稳定读取受限 Markdown。
- Compact TASK 比当前常用 A/B TASK 更长或需要更多重复填写。
- Legacy Reader 对真实样本产生不可接受的误判。
- TASK 和 TASK_BOARD 无法建立清晰单向权威关系。
- Overlay 需要任意深度 merge 或通用插件注册器才能工作。
- GPT‑5.6 Adapter 影响确定性 diagnostics。
- v0.6 发布边界仍未收口，导致无法判断兼容基线。

回滚原则：删除新 Reader、lint 和 v0.7 模板即可；旧文档、旧 TASK 和旧工作流必须继续可用，不做数据迁移回滚。

## 20. 已确认的三项决策

用户于 2026-07-11 回复“按推荐方案执行”，确认以下推荐选择：

1. **Canonical 格式**：采用 `## Workflow Contract` 受限 Markdown 键值块，不使用 YAML frontmatter 或 JSON sidecar。
2. **v0.7 首批范围**：只做 Contract、fixture、Legacy/v0.7 Reader、只读 lint、Compact Template、最小 SKILL/PROMPTS Writer 同步和 TASK_BOARD drift；不把 GPT‑5.6 Adapter 或外部同步列为首发门禁。
3. **发布顺序**：先完成 `REL-001`，把 `0.6.0 Unreleased` 与当前 `0.5.2` 的关系收口，再开始 v0.7 实现。

已据此创建 `REL-001` 和 `CONTRACT-001`–`CONTRACT-006` 的正式 TASK；本次确认只授权任务拆分，仍需逐项进入 `execute_task`，不一次性大改。

## 21. 本 RFC 的完成边界

本文件完成的是方案收敛，不代表以下事项已完成：

- v0.6 已发布。
- v0.7 Contract 已生效。
- 模板重复字段已修复。
- Validator、fixture 或 CI 已实现。
- GPT‑5.6 API 已接入。
- 任何 TASK 已迁移。

三项决策已确认。下一步只有在用户明确指定 `执行 REL-001` 或等价指令后，才进入首个任务；本 RFC 获批不等于批准 tag、merge、push、发布或后续所有任务自动执行。
