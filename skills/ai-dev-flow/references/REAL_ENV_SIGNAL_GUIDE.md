# 实机测试信号复现（real_env_signal）指南

实机测试信号复现（`real_env_signal`）用于把用户真实环境、真实设备、真实数据或本机特殊环境中的失败反馈，转换成可重复触发、可判定失败、修复后可判定通过的信号。它不是自动控制真实环境，也不等于 agent 已完成实机测试。

## 边界

- 不是新任务状态。
- 不替代 TASK、Acceptance Feedback Gate、Bug 诊断（`bug_diagnosis`）或审查-修复循环（`review_repair_loop`）。
- 不承诺自动脚本、自动实机控制、自动 GitHub 同步、自动多 agent 调度或自动发布。
- 没有 RED 失败信号时，不得盲目修复。
- 没有用户回传证据时，只能记录待确认或阻塞（Blocked）建议。

## 复现能力分类

1. agent 可自动复现：agent 能在当前环境运行命令、测试、页面或样例。
2. 样例文件 replay：用户可提供脱敏样例文件、输入、输出或日志，agent 可在本地 replay。
3. 用户实机 HITL：agent 无法亲自复现，需要用户按步骤在真实环境执行并回传证据。
4. 暂无法复现：只能增加最小诊断日志或列出补充信息，不能直接猜测修复。

## RED / GREEN / SIGNAL

- RED 失败信号：用户当前能稳定观察到的失败，例如报错文本、错误码、错误输出、截图状态、生成文件差异或性能阈值。
- GREEN 通过信号：修复后应观察到的通过条件，例如无报错、输出一致、关键流程完成或指定日志消失。
- SIGNAL 证据来源：命令输出、测试日志、fixture、截图、录屏、CSV、生成文件、用户操作步骤、真实环境回传记录。

## 用户实机 HITL 步骤

HITL 步骤应短、可执行、可回传：

1. 准备环境和输入。
2. 执行最小步骤。
3. 观察 RED 或 GREEN。
4. 回传日志、截图、录屏、输出文件摘要或脱敏样例。
5. 标记是否必现、偶现或已消失。

## 临时诊断日志规则

- 临时日志必须有唯一前缀，例如 `[ADF-SIGNAL:TASK-123]`。
- 只记录判断 RED/GREEN 所需的最小字段。
- 不记录密钥、Token、真实账号、个人隐私或完整私有数据。
- 修复后必须清理临时日志，或明确转为正式日志并经过确认。

## 用户回传证据模板

```markdown
## 用户实机回传证据

- TASK：
- 验收 UA：UA4 / UA5 / UA6 / UA7
- 环境：
- 输入 / 样例：
- 操作步骤：
- RED 是否复现：是 / 否 / 偶现 / 待确认
- GREEN 是否成立：是 / 否 / 待确认
- 实际结果：
- 证据位置：日志 / 截图 / 录屏 / CSV / 输出文件 / 其他
- 是否可提供脱敏样例：是 / 否 / 待确认
```
