# 测试先行（tdd_task）指南

测试先行（`tdd_task`）用于在实现或修复前先建立行为级反馈信号。它不是新任务状态；任务仍应保持草稿（Draft）、可执行（Ready）、执行中（In Progress）、待审查（Review）等状态机流转。

## 适用场景

- 用户明确要求 TDD、测试先行或 RED GREEN。
- bug 已有清晰复现路径。
- 复杂逻辑、边界条件或回归风险较高。
- 修复需要证明用户描述的 RED 已经消失。
- 代码已有测试框架或可通过公开接口构造测试。

## 不适用场景

- 当前项目没有测试框架，且建立测试缝隙会明显扩大范围。
- 问题只能在真实环境中复现，暂时只能使用用户实机 HITL 或样例 replay。
- 纯文档任务、流程模板整理或只读审查。
- 用户明确要求先只做诊断，不写测试。

## 原则

- 测试验证行为，不验证私有实现细节。
- 优先覆盖公开接口、命令、UI 可观察输出、日志、文件输出或 API 响应。
- 一个行为对应一个失败测试，再做最小实现让它通过。
- 红灯（RED）阶段不重构；绿灯（GREEN）后才考虑小范围整理。
- 没有自动测试条件时，必须写清替代验证方式和原因。
- 不为通过测试而改变业务含义或降低断言强度。

## RED / GREEN 循环

1. 选择一个可观察行为。
2. 写失败测试或最小复现命令，确认 RED。
3. 做最小实现或修复。
4. 运行同一测试，确认 GREEN。
5. 运行相关回归测试或替代验证。
6. 更新任务文件，记录测试名称、命令、失败信号、通过信号和未覆盖风险。
