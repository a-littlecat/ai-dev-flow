# 通用代码审查清单

本清单用于审查代码任务。审查线程只做审查和判断，不直接改代码。结论应写回任务文件或项目约定的审查记录中。

## 审查严重等级

- P0：阻止合并或验收，必须立即修复。
- P1：高风险，验收前必须修复。
- P2：建议修复，可单独开后续任务。
- P3：风格或可选建议，不阻止验收。

## 审查原则

- 只审查当前任务范围。
- 不把历史问题全部压到当前任务。
- 发现历史问题时记录为后续任务。
- 审查线程只审查，不直接修复。
- 审查结论必须写入任务文件。
- 审查结论必须写入当前任务文件的“代码审查”和“Diff 审查”段落，或项目约定的审查记录。
- 只在聊天中输出审查意见不算完成审查。
- 审查线程可以更新审查记录和任务看板 Review 状态，但不得修改业务代码。
- 审查必须明确是否允许进入验收建议和对应用户动作等级。
- Review Hub 审查 Batch / Wave 时，必须逐任务输出结论，不得只给批次或波次总论。
- Batch / Wave 中每个任务都必须有独立 diff 范围、验证结果和 UA 等级。

## 0. Git / Diff 基线检查

- 是否记录任务等级。
- 是否按等级选择了正确执行方式。
- 是否记录 base commit。
- 是否能生成 diff。
- diff 是否只包含当前任务相关文件。
- 是否存在未提交改动。
- 是否存在未跟踪但应提交的文件。
- 是否存在不应提交的文件。
- 是否执行 `git diff --check`。
- 是否明确本次审查范围。
- 小任务是否避免了不必要的分支复杂度。
- 大任务是否使用分支或 Worktree。
- Batch 中每个任务的 diff 是否可拆分。
- Wave 中是否已检查文件锁、模块锁和依赖关系。
- 是否存在多个任务共享同一模糊 diff 的风险。

## 1. 任务范围

- 是否只完成当前任务。
- 是否改了无关文件。
- 是否引入额外重构。
- 是否修改了任务明确禁止修改的范围。
- 是否把顺手优化伪装成任务必要改动。

## 2. 功能正确性

- 是否满足任务完成标准。
- 是否保留旧功能。
- 是否有边界情况遗漏。
- 是否处理了失败路径。
- 是否和现有输入、输出、配置或接口约定一致。

## 3. 架构和分层

- 是否破坏模块边界。
- 是否把临时代码写进核心模块。
- 是否把 UI、业务逻辑、基础设施混在一起。
- 是否引入不必要的全局状态。
- 是否复制粘贴了本应复用的逻辑。
- 是否新增了没有必要的大型依赖。

## 4. 可靠性

- 错误处理是否清晰。
- 空值处理是否充分。
- 路径处理是否兼容不同运行环境。
- 日志和用户提示是否准确。
- 是否存在兼容性风险。
- 是否存在并发、重试、超时或资源释放风险。
- 是否有可能造成数据丢失或不可逆修改。

## 5. 安全与隐私

- 是否泄露密钥、Token、账号、密码或证书。
- 是否提交本机绝对路径。
- 是否提交私有文件、临时文件或个人配置。
- 是否引入危险命令。
- 是否扩大了文件系统、网络、权限或执行范围。
- 是否把敏感信息写入日志、错误提示或测试快照。

## 6. 构建和验证

- 是否能构建。
- 是否说明测试方式。
- 是否运行了任务要求的验证命令。
- 是否需要人工验证。
- 是否说明了未能验证的原因。
- 是否有必要增加或更新测试。

## 7. 审查结论格式

建议按以下格式输出：

```markdown
## 审查结论

结论：通过 / 需要修改 / 不建议合并

是否允许进入验收建议：是 / 否

建议用户动作等级：UA0 / UA1 / UA2 / UA3 / UA4 / UA5 / UA6 / UA7 / 待确认

最高严重等级：P0 / P1 / P2 / P3 / 无

## 必须修改项

- 无

## P0 / P1 必须修改项

- 无

## P2 建议修改项

- 无

## P3 可选建议

- 无

## 风险

- 无

## 推荐验证步骤

1. 待填写。

## 说明

简短说明审查依据，包括任务文件、主要 diff、已查看的验证结果。
```

## 8. Batch / Wave 审查补充

审查 Batch 或 Wave 时，输出必须包含：

- 批次或 Wave 编号。
- 每个任务的独立结论。
- 每个任务的 P0/P1/P2/P3。
- 每个任务的 UA 等级。
- 每个任务是否允许进入验收建议。
- 必须单独复审或移出 Batch / Wave 的任务。

不得只写“Batch 通过”或“Wave 通过”。如果 diff 无法按任务拆分，应标记为范围风险，并要求拆分后复审。

### 结论含义

- `通过`：未发现阻止合并的问题，但仍需按项目规则完成对应用户动作等级和用户确认。
- `需要修改`：存在必须修改项，修复后需要重新审查或补充验证。
- `不建议合并`：存在范围错误、严重缺陷、验证不足或高风险改动。

### 必须修改项

只列会阻止任务完成或合并的问题，例如：

- 功能不满足完成标准。
- 破坏旧功能。
- 修改无关模块。
- 引入安全隐患。
- 缺少必要验证。

### 建议修改项

列不阻止当前任务完成，但值得后续优化的问题，例如：

- 命名可以更清楚。
- 文档可以更完整。
- 可读性可以提高。
- 后续可以补充测试。

### 风险

说明即使审查通过仍需要用户知道的风险，例如：

- 未能在目标环境验证。
- 依赖外部服务或环境配置。
- 存在兼容性不确定性。
- 人工体验验收仍未完成。
