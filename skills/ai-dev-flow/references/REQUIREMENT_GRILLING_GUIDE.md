# 需求拷问（requirement_grilling）指南

需求拷问（`requirement_grilling`）用于需求模糊、高风险、容易返工或影响范围不清时，把关键问题问清楚。它发生在创建任务（`create_task`）或规划任务（`plan_task`）前后，不直接执行代码。

## 使用原则

- 先读项目文档和相关代码；能查清的问题不要问用户。
- 只问会阻塞任务边界、验收标准、风险决策或不可逆动作的问题。
- 交互式 agent 一次只问一个最关键问题。
- 非交互式或 Codex 场景可输出阻塞问题清单，并给出推荐默认假设。
- 非阻塞模糊点写入 Intake、风险或任务假设，不要卡住整个流程。
- 需要进入 `repair_task`、扩大范围、创建新 TASK、关闭任务或改变已确认状态前必须用户确认。

## 适用场景

- 需求只描述“优化”“完善”“修一下”，没有可观察完成标准。
- 涉及真实业务流程、用户数据、外部系统、架构、依赖或发布。
- 同一任务可能拆成多个互相独立的 TASK。
- 用户反馈和现有任务范围可能不一致。

## 输出模板

```markdown
## 需求拷问记录（requirement_grilling）

- 原始需求：
- 当前模式：创建任务（create_task）/ 规划任务（plan_task）
- 已读取文件：
- 已查清事实：

### 阻塞问题

1. 问题：
   - 为什么阻塞：
   - 推荐默认假设：
   - 用户不回答时的安全处理：

### 非阻塞模糊点

- 模糊点：
- 记录位置：Intake / TASK / 风险 / 后续任务

### 建议下一步

- 继续 create_task / 进入 plan_task / 等待用户确认 / Blocked
```
