# 从 v0.7 迁移到 v0.8

v0.8 改的是“什么时候启用 Skill”，不是历史 TASK 的数据格式。旧项目可以渐进使用，不需要一次性迁移。

## 用户最多做 3 步

1. 更新 `skills/ai-dev-flow/` 包；不要批量改历史 TASK。
2. 从 `AGENTS_COMPAT.md` 选择真正需要的最小规则，合并进项目现有 `AGENTS.md`，不要覆盖项目规则。
3. 新请求先路由：低风险任务 `DoNotUseSkill`；需要留证用 Tracked；高风险或交付动作使用 Controlled。

## 保持兼容

- 旧 TASK、Full/Legacy 和 v0.7 Compact TASK 原样保留。
- Contract schema 继续是 `adf/v0.7.0`；Skill 包 `0.8.0` 不要求 `adf/v0.8.0`。
- Reader、`workflow_lint` 和 TASK_BOARD drift 仍是只读工具，不自动修复或迁移。
- `TASK_TEMPLATE_COMPACT.md` 仅为 v0.7 兼容；v0.8 Lite 不建 TASK，新 Tracked / Controlled 使用 `TASK_TEMPLATE.md`。
- 不增加 required dependency，不需要数据库、服务或模型供应商配置。

## 行为变化

| v0.7 常见做法 | v0.8 默认做法 |
|---|---|
| A/B 新任务创建 Compact TASK | 满足 Lite 条件时退出 Skill，不建 TASK |
| 默认读取多份工作流说明 | 默认只读 `SKILL.md + CORE.md` |
| 多种 Loop/角色/Batch/Wave 进入主说明 | 只在用户明确需要时读取对应兼容指南 |
| 普遍考虑独立 Reviewer | Tracked 风险触发，Controlled enforcement point 强制 |
| repair 固定两轮后人工接管 | 基础 2 轮；严格收敛时允许且仅允许第 3 轮 |

## 回滚

回滚 v0.8 入口只需恢复上一版 Skill 包；历史 TASK 和 v0.7 Contract 无需回滚。`evaluations/v0.8/**` 与 `prototypes/v0.8-lite/**` 是审计证据，不应因迁移或回滚而改写。

## 发布边界

仓库中 `VERSION=0.8.0` 只表示实现身份。没有 tag、push 或 Release 证据时，应写“0.8.0 已实现、尚未发布”，不能写“已发布”。
