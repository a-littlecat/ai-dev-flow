# 用户动作等级 / 验收指南

本指南用于把“人工验收”拆成清晰的用户动作等级。任务完成后，agent 不得笼统写“需要人工验收”，必须说明用户到底要做什么。

人工验收不是人工代码审查。用户主要验证可观察行为、关键流程、输出结果、错误提示、回归影响和是否符合需求；代码质量、diff 风险和范围越界主要由 AI 审查、自动验证和 diff 审查辅助判断。

## UA0：无需用户验收，agent 自证即可

适用：

- 纯格式整理。
- 模板字段补充。
- `CHANGELOG` / `VERSION` 同步。
- 任务状态修正。
- 无业务影响的文档搬运。
- 无代码影响的提示词整理。

要求：

- agent 必须提供修改文件清单。
- agent 必须说明为什么无需用户验收。
- agent 必须记录验证方式或替代检查。
- 不得把有业务影响的任务标为 UA0。

用户动作：

- 无需操作，只查看摘要即可。

关闭规则：

- UA0 默认不自动标记为 `Closed`。
- agent 可以写“建议关闭”，但不得擅自把任务状态改为 `Closed`。
- 只有用户明确确认，或项目规则已明确授权 UA0 可自动关闭时，agent 才能关闭任务。

## UA1：用户只看摘要

适用：

- `TASK_BOARD` 更新。
- 任务编号整理。
- 轻量 `README` 补充。
- 小范围文档修订。
- 无代码影响的流程整理。

用户动作：

- 看一眼摘要。
- 确认没有明显偏离目标。

## UA2：用户需要读文档 / 方案

适用：

- 架构方案。
- 迁移 RFC。
- 商业化方案。
- 任务拆分方案。
- `AGENTS.md` 规则。
- Skill 工作流规则。
- 重要提示词模板。

用户动作：

- 判断方向是否正确。
- 判断是否太复杂。
- 判断是否符合自己的工作习惯。
- 不需要实机测试。

## UA3：用户只看验证证据，不自己运行

适用：

- agent 能自己构建。
- agent 能自己跑测试。
- agent 能提供截图、日志、diff 审查结果。
- 内部重构但行为不变。
- 不依赖用户本机特殊环境的小改动。

用户动作：

- 看构建 / 测试 / diff / 截图 / 日志证据。
- 判断是否允许进入下一步。
- 通常不需要用户自己运行。

## UA4：用户需要本地运行验收

适用：

- 普通前端页面。
- 本地 CLI 工具。
- 本地桌面程序。
- 本地 UI 宿主程序。
- 嵌入式 Web 页面。
- 不依赖外部真实业务环境的功能。

用户动作：

- 在本机运行。
- 点击关键按钮。
- 检查是否报错。
- 检查是否符合可观察结果。

## UA5：用户需要实机业务测试

适用：

- 外部软件插件。
- 外部宿主软件插件。
- 真实设备。
- 真实图纸、真实数据、真实账号。
- 批处理输出。
- 文件导出。
- 本机环境依赖强的功能。
- agent 无法完整复现用户环境的功能。

用户动作：

- 在真实软件或真实环境中测试。
- 使用真实样例数据。
- 检查输出结果。
- 检查错误提示和异常路径。

## UA6：用户需要回归验收

适用：

- 核心流程修改。
- 重构。
- 通信协议修改。
- 核心服务修改。
- 文件路径 / 输出逻辑。
- 登录、授权、计费。
- 数据写入、删除、迁移。
- 任何可能破坏旧功能的改动。

用户动作：

- 运行关键旧流程。
- 确认旧入口仍可用。
- 确认典型用例没有坏。
- 记录回归结果。

## UA7：必须用户决策，不能由 agent 接受

适用：

- 是否合并。
- 是否删除旧实现。
- 是否切换技术栈。
- 是否新增大依赖。
- 是否改变架构。
- 是否修改授权、登录、收费逻辑。
- 是否修改发布流程。
- 是否执行不可逆操作。
- 是否接受明显风险。

用户动作：

- 做负责人决策。
- 明确同意或拒绝。
- agent 不得代替用户判断。

## 默认选择规则

速查：UA0（无需用户验收）、UA1（用户看摘要确认）、UA2（用户读文档或方案）、UA3（用户看验证证据）、UA4（用户本地运行）、UA5（真实环境或实机业务测试）、UA6（核心流程或回归验收）、UA7（用户决策或高风险确认）。

- 文档任务默认 UA0 / UA1 / UA2。
- 方案任务默认 UA2。
- 有自动测试且无用户可见变化的代码任务默认 UA3。
- UI 或本地可运行功能默认 UA4。
- 依赖真实软件、真实设备、真实数据、真实账号的任务默认 UA5。
- 核心流程、协议、数据、输出、路径、登录、授权、发布相关任务默认 UA6。
- 架构、技术栈、合并、删除、发布、不可逆操作默认 UA7。
- 如果无法判断，写“待确认”，不要默认要求用户实机测试，也不要默认跳过验收。
- 用户动作等级只描述用户需要做什么，不等于任务状态；即使是 UA0，也默认只建议关闭，不自动 `Closed`。
- Batch / Wave 中每个任务都必须单独判断 UA 等级；不得只给批次或波次一个总 UA。

## 禁止误判

- 不得把代码任务随便标为 UA0。
- 不得把依赖真实环境的任务标为 UA3。
- 不得把核心流程修改只标为 UA4。
- 不得用“已构建通过”替代业务验收。
- 不得用“代码看起来没问题”替代用户可观察行为。
- 不得让用户逐行审代码作为验收方式。
- 人工验收不是人工代码审查。
- UA5 / UA6 / UA7 任务默认不得进入代码并行；如需例外，必须用户明确确认。

## 验收失败处理规则

用户反馈 UA4 / UA5 / UA6 / UA7 验收失败时，agent 不得直接进入代码修改。

必须先执行“验收失败反馈闸门”，做只读诊断和分类：

- 原任务完成标准未满足：当前任务建议转为 `Needs Fix`，进入 `review_repair_loop`。
- 本轮修改引入回归：当前任务建议转为 `Needs Fix`；严重等级至少 P1，必要时升级为 UA6 回归验收。
- 新需求或范围扩大：当前任务不得顺手修，建议创建新 TASK。
- 用户环境 / 配置 / 权限 / 数据 / 外部设备问题：当前任务建议转为 `Blocked`，并列出最小补充信息。
- 证据不足：只输出诊断问题清单，不修改代码，不进入 `repair_task`。

如果任务已经 `Accepted` 或 `Closed` 后才收到失败反馈，默认创建回归修复 TASK 或等待用户明确授权；不得直接把原任务乱跳回 `Review`、`Needs Fix` 或其他状态。

修复限制：

- 默认最多 2 轮 repair。
- 每轮 repair 前必须说明本轮假设、证据、拟修改文件和验证方式。
- 每轮 repair 后必须重新进入 `review_task`。
- 两轮后仍未通过，必须停止并建议人工接管或重新拆任务，不得继续猜测修改。
- 如果无法在 agent 环境中复现，必须补充日志、诊断命令、mock、截图说明或用户实机复测步骤。

验收失败反馈必须记录到 TASK 文件的“用户验收反馈 / 实机测试反馈”区块；只在聊天中处理不算完成。

## 输出格式

每个任务完成后必须输出：

```markdown

用户动作等级：UA0（无需用户验收）/ UA1（用户看摘要确认）/ UA2（用户读文档或方案）/ UA3（用户看验证证据）/ UA4（用户本地运行）/ UA5（真实环境或实机业务测试）/ UA6（核心流程或回归验收）/ UA7（用户决策或高风险确认）/ 待确认

是否需要用户实机测试：是 / 否 / 待确认

用户需要做什么：
1. 待填写

agent 已提供的证据：
- 修改文件：
- diff 摘要：
- 构建 / 测试：
- 截图 / 日志：
- 审查结论：

可跳过实机测试的原因：
- 待填写

不验收的风险：
- 待填写

是否允许关闭任务：
建议关闭 / 是（用户已确认） / 是（项目规则已授权） / 否 / 待确认
```
