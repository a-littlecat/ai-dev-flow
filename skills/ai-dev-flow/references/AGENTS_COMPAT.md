# 可合并到项目 AGENTS.md 的通用规则

以下内容可按需合并到项目根目录 `AGENTS.md`。合并前请保留项目已有规则，并删除与项目不适用或重复的部分。

## 默认语言和文档

- 默认使用中文沟通。
- 默认使用中文编写项目文档。
- 代码标识符、命令、路径、配置项、API 字段名和第三方库名称保持英文。

## 任务开始前

- 先读取项目已有规则和关键文档，包括 `AGENTS.md`、`README.md`、`docs/PROJECT_INDEX.md`、`docs/TASK_BOARD.md` 和当前任务文件。
- 开始前判断当前工作模式：init_project、create_task、plan_task、execute_task、review_task、repair_task、close_task 或 status_report。
- 不同模式不得混用。
- 如果需求模糊，先进入 Intake；Intake 不是 TASK，不直接执行代码。
- 如果使用 Loop，必须声明 Loop 类型和当前子模式；Loop 不是任务状态。
- 当前轮输出应声明角色：Orchestrator、Planner、Engineer、Reviewer、Verifier、Repairer 或 Archivist。
- 所有任务必须有明确状态。
- 不确定状态不得猜测，必须写“待确认”或标记为 `Blocked`。
- 如果任务文件不存在，先建议或创建任务文件，不要只依赖聊天记录。
- 如果需求过大或边界不清，先写 plan 或 RFC，再拆成小任务。
- 大任务先写计划。
- 完成标准必须可验证。
- 没有验证方式的任务不能标记为验收通过。
- 代码任务开始前必须检查 Git 状态。
- 没有 Git baseline 不得开始代码任务。
- 代码任务必须记录 base commit。
- 任务开始前必须判断任务等级。
- 工作区已有未提交改动时，必须先判断改动来源和归属。
- 修改前先简短说明目标、拟修改范围、风险和验证方式。

## 任务执行

- 小任务不强制分支。
- 中等任务建议分支。
- 大任务必须使用分支或 Worktree。
- A/B 小任务可以批量执行和批量审查。
- C/D 任务默认单独执行和单独审查。
- Batch 是一个执行会话顺序处理多个小任务，不是并行写代码。
- Wave 是多个执行会话同时处理多个互不冲突任务。
- 批量执行前必须确认任务之间不冲突。
- 并行执行前必须声明预计修改文件和影响模块。
- 文件或模块冲突时不得并行。
- Parallel Wave 中代码任务默认必须使用独立分支或 Worktree。
- 多个代码执行会话不得默认共享同一工作区。
- 如果无法确认工作区隔离，不得启动代码并行。
- B 级代码 Batch 必须能按任务拆分 diff；否则不得批量审查通过。
- 多个 B 级任务修改同一文件时，默认不进入同一 Batch。
- D 级任务默认不得和其他代码任务并行。
- UA5 / UA6 / UA7 任务默认不得进入代码并行。
- ai-dev-flow 不禁止 harness 或模型自动使用 subagents；只读 subagents 默认允许，写代码 subagents 必须受控，并行写代码 subagents 默认需要用户确认。
- 不要跨任务修改无关模块。
- 不要顺手做无关重构。
- 任务膨胀时必须停止并拆分。
- 不要擅自新增大型依赖、替换技术栈、改变数据结构或修改发布流程。
- 需要用户确认的操作不得擅自执行。
- Reviewer 不得直接修复。
- Engineer 不得自我批准。
- Repairer 只处理审查指出的问题。
- Orchestrator 默认只做总览、分拣和状态报告，不直接改业务代码。
- Verifier 只验证和整理证据，不扩大修改范围。
- 不要提交密钥、Token、证书、私有配置、个人文件或本机敏感信息。
- 不得提交构建产物、依赖目录、日志、本机配置或密钥文件。
- 不得盲目执行 `git add .`。
- 发现旧文档和代码不一致时，指出冲突，不要编造现状。

## 任务状态

- 完成后更新当前任务文件和 `docs/TASK_BOARD.md`。
- 任务文件应记录修改文件、执行结果、验证命令、验证结果、遗留问题和下一步建议。
- 任务完成后必须写明用户动作等级：UA0 / UA1 / UA2 / UA3 / UA4 / UA5 / UA6 / UA7 / 待确认。
- 不得笼统写“需要人工验收”。
- 不得默认要求用户实机测试。
- 不得默认跳过用户验收。
- UA0 / UA1 任务可只给摘要。
- UA0 不等于自动 `Closed`；默认只能建议关闭，除非用户或项目规则授权。
- UA2 任务需要用户阅读文档或方案。
- UA3 任务需要 agent 提供验证证据。
- UA4 / UA5 / UA6 任务需要用户运行、本地测试、实机业务测试或回归验收。
- UA7 任务必须用户明确决策。
- 用户不需要逐行审代码。
- 不要把聊天记录作为唯一状态来源。
- Memory 只记录稳定、可复用项目知识；不得写入聊天全文、密钥、本机路径、私有账号或未确认猜测。
- 如果任务涉及长期规则，应检查 `PROJECT_CONSTITUTION.md`；违反 MUST / MUST NOT 时应标记 P0/P1。
- 如果无法完成或无法验证，明确写出原因和阻塞点。

## 代码审查

- 代码任务必须审查。
- Batch / Wave 可由 Review Hub 集中审查，但必须逐任务输出结论。
- 不得因为 Batch / Wave 通过就默认每个任务都通过。
- 不得让多个任务共享一个模糊 diff。
- 每个批量任务和并行任务仍必须独立记录 base commit、diff、验证结果和用户动作等级。
- 不得因为批量或并行跳过 Git baseline、diff 审查、用户动作等级和任务文件更新。
- 审查线程只审查，不直接改代码。
- 审查必须基于任务文件、base commit、明确 diff、验证记录和代码审查清单。
- 审查线程只审查，不修复。
- 修复线程只处理审查指出的问题。
- Review-Repair Loop 默认最多 2 轮 repair；两轮后仍有 P0/P1 时必须停止并交给用户。
- 审查结论必须明确为：通过、需要修改或不建议合并。
- 审查结论必须写回当前任务文件的“代码审查”和“Diff 审查”段落，或项目约定的审查记录。
- 只在聊天中输出审查意见不算完成审查。
- 审查线程可以更新审查记录和 `docs/TASK_BOARD.md` 的 Review 状态，但不得修改业务代码。
- 未通过审查时，不得建议合并。
- 审查必须明确是否允许进入验收建议和对应用户动作等级。

## 合并和危险操作

- 未经用户明确确认，不得合并。
- 未经用户明确确认，不得 push、release、reset、rebase、force push、删除分支、删除 Worktree 或删除文件。
- 不得自动创建、关闭或同步 GitHub Issue；GitHub Issues backend 在 v0.6.0 只是可选设计和 mapping preview。
- 不得声称当前流程具备自动 subagent 调度或自动多 agent 调度能力。
- subagents 不得自动 merge、push、release、delete、reset、rebase、删除分支、删除 Worktree 或 GitHub 同步。
- 如果当前 harness 不支持 Worktree，不得启动 Parallel Wave 代码任务；如果不支持 subagents，应使用普通总览 / 执行 / 审核会话流程。
- 不得在工作区混乱时建议合并。
- 合并前必须确认任务完成标准、审查状态、用户动作等级、必要验收或决策状态和剩余风险。

## 输出要求

- 输出要简洁，优先给结论、步骤和可执行信息。
- 修改完成后说明：
  - 修改了哪些文件。
  - 每个文件主要改了什么。
  - 执行了哪些验证命令。
  - 是否还有风险或未完成项。
  - 下一步建议。

## 扫描和搜索排除

搜索项目文件时，不要扫描构建产物、依赖目录和日志目录。常见排除项包括：

- `.git/`
- `node_modules/`
- `vendor/`
- `dist/`
- `build/`
- `out/`
- `bin/`
- `obj/`
- `.next/`
- `.cache/`
- `coverage/`
- `logs/`
- `*.log`
