# 批量小任务 Batch 指南

Batch 用于把多个低风险 A/B 小任务放进同一个执行会话中顺序完成，并交给一个审核会话批量审查。Batch 不是并行，不能吞掉任务边界。

## 核心原则

- `TASK` 仍然是最小记录单位。
- TASK 是最小责任单位。
- Batch 是执行和审查打包方式，不是新的任务粒度。
- A/B 小任务可以批量执行、批量审查。
- C/D 任务默认不得进入批量执行。
- 批量执行是在同一个执行会话中顺序完成一批低风险小任务。
- 批量审查必须按任务逐项输出结论，不得只给一个总评价。
- 每个任务仍需记录 base commit、diff 范围、验证结果和用户动作等级。
- 批量任务仍需保持 Git diff 清晰、可回滚、可审查。
- A 级文档 Batch 可以使用一个 commit。
- B 级小代码 Batch 推荐每个 TASK 单独 commit。
- 如果 B 级小代码 Batch 不单独 commit，必须在 BATCH 文件中记录每个 TASK 的修改文件、diff 归属和验证结果。
- 多个 B 级任务如果会修改同一文件，默认不得进入同一个 Batch，应拆成单任务执行；用户明确确认后也必须标记风险。
- 如果 diff 无法按任务拆分，必须停止批量审查，标记“diff 归属不清”，并建议拆分任务或拆分 commit。
- 不得因为 Batch 通过就默认批次内每个任务都通过。

## 可批量执行

### A 级文档小任务

适用：

- `README` 小范围修改。
- `TASK_BOARD` 更新。
- `TASK_TEMPLATE` 字段补充。
- `CHANGELOG` / `VERSION` 同步。
- `PROMPTS` 小范围修改。
- `AGENTS_COMPAT` 小范围补充。
- 无业务代码影响的流程文档整理。

推荐：

- 一次批量 3~8 个 A 级任务。
- 一个执行会话顺序完成。
- 一个审核会话批量轻量审查。
- 通常使用 UA0 / UA1 / UA2。
- 通常不需要用户实机测试。

### B 级小代码任务

适用：

- 小文案。
- 小样式。
- 单点空值判断。
- 局部小 bug。
- 不涉及核心流程的小范围修改。

推荐：

- 一次批量 2~3 个 B 级任务。
- 不建议跨多个核心模块。
- 必须记录 base commit。
- 必须保持 diff 小而清晰。
- 推荐每个 TASK 单独 commit。
- 如果不单独 commit，必须记录每个 TASK 的修改文件、diff 归属和验证结果。
- 多个 B 级任务修改同一文件时，默认拆成单任务执行或等待用户确认。
- 必须逐任务给出审查结论和 UA 等级。

## 不可批量执行

以下任务默认必须单独执行、单独审查：

- C 级中等代码任务。
- D 级大任务或高风险任务。
- 架构迁移。
- 技术栈迁移。
- 通信协议。
- 核心服务。
- 项目入口或 UI host 启动链路。
- 前端全局状态流。
- 构建 / 发布 / 依赖。
- 数据写入、删除、迁移。
- 外部软件插件。
- 真实设备、真实数据、真实账号相关任务。
- 批处理输出、文件导出、核心业务流程。
- 任何 UA5 / UA6 / UA7 任务。
- diff 无法按任务拆分的任务。
- agent 无法判断风险等级的任务。

## 批次限制

默认建议：

- A 级任务：一次最多 8 个。
- B 级任务：一次最多 3 个。
- A+B 混合：一次最多 5 个。
- C/D：不进入批次。

如果项目规则更严格，以项目规则为准。超过限制时，agent 应建议拆成多个 Batch。

## Batch 文件格式

可选创建 `docs/batches/BATCH-xxx.md`：

```markdown

## 批次类型

## 执行位置

- 主项目 / 分支 / Worktree

- 当前 HEAD：
- Diff 范围：
- 是否有未提交改动：
```

## 批量执行流程

1. 总览会话从 `TASK_BOARD` 中挑出候选任务。
2. 只选择 A/B 小任务。
3. 检查任务之间是否冲突。
4. 检查是否会修改同一文件或同一核心模块。
5. 多个 B 级代码任务修改同一文件时，默认拆成单任务执行或等待用户确认。
6. 记录 base commit。
7. 可选创建 `BATCH-xxx.md`。
8. 执行会话顺序完成批次内任务。
9. B 级小代码任务推荐每个 TASK 单独 commit；如不单独 commit，记录 per-task 修改文件、diff 归属和验证结果。
10. 每个 `TASK` 文件仍需更新执行结果、修改文件、验证结果、diff 范围和用户动作等级。
11. 批次完成后生成批量审查交接包。
12. 交给固定审核会话批量审查。

## 批量审查流程

要求：

- 只用于 A/B 小任务。
- 每个任务单独审查。
- 每个任务单独输出结论。
- 每个任务单独给 UA 等级。
- 发现 P0/P1 的任务必须拉出单独复审。
- 发现 C/D 风险时必须停止批量审查。
- 如果 diff 无法按任务拆分，必须停止批量审查，标记“diff 归属不清”，并建议拆分任务或拆分 commit。
- 审查结论写回每个 `TASK` 文件。
- 可选更新 `BATCH` 文件和 `TASK_BOARD`。

```markdown
## 批量审查结论

批次：BATCH-xxx
整体结论：通过 / 部分通过 / 需要拆分 / 不建议合并

## 逐任务结论

### TASK-001

- 等级：
- Diff 范围：
- 修改文件：
- Per-task commit：
- Diff 归属：清晰 / 不清 / 不适用
- 是否范围越界：
- P0/P1：
- P2/P3：
- 审查结论：通过 / 需要修改 / 不建议合并
- 建议用户动作等级：
- 是否需要实机测试：
- 是否允许进入验收建议：

### TASK-002

- 无

- 无

- 无
```
