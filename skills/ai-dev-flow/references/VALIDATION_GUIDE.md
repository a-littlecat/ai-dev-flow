# 验证指南

任务完成标准必须可验证。没有验证方式的任务不能标记为 Accepted。

验证和验收不是同一件事：验证是证明任务完成的证据，验收是用户或指定验收人决定是否接受结果。

## 验证类型

- 文档检查。
- 静态检查。
- 构建验证。
- 单元测试。
- 集成测试。
- 手动 smoke test。
- UI 截图验证。
- 日志验证。
- 回归验证。
- 不适用。

## 完成标准规则

- 完成标准必须可验证。
- 不接受“优化体验”“完善功能”这种模糊描述。
- 每个任务必须写清楚如何证明完成。
- 如果无法自动测试，必须写人工验证步骤。
- 如果没有验证，任务不能标记为 Accepted。
- 验证失败时不得声称任务完成。
- 不是所有验收都需要实机测试。
- 不是所有任务都需要用户验收。
- 文档任务可以使用 UA0 / UA1 / UA2。
- 证据充分且无用户可见变化的任务可以使用 UA3。
- UI、本地程序、插件、真实业务流程需要 UA4 / UA5 / UA6。
- 不能笼统写“需要人工验收”，必须写用户动作等级。
- 用户不需要逐行审代码。
- Batch / Wave 中每个任务都必须有独立验证结果；不得用批次或波次总验证替代单任务验证。
- bug 修复验证不等于“构建通过”；必须证明用户描述的 RED 失败信号已经消失。
- 实机问题可使用用户实机 HITL 复测作为验证方式，但必须写清步骤、回传证据和判断标准。
- 对外部软件、真实设备、真实文件输出、真实账号或真实数据任务，agent 必须明确哪些验证只能由用户完成。

## 验证与用户动作等级

- UA0：agent 自证即可，但必须提供修改文件、验证方式或替代检查；默认只能建议关闭任务，不自动 `Closed`。
- UA1：用户只看摘要。
- UA2：用户需要读文档或方案。
- UA3：用户只看验证证据，不自己运行。
- UA4：用户需要本地运行验收。
- UA5：用户需要实机业务测试。
- UA6：用户需要回归验收。
- UA7：必须用户决策，不能由 agent 接受。

更多规则见 `ACCEPTANCE_GUIDE.md`。

## Batch / Wave 验证

- Batch 可以共用一次执行会话，但每个任务必须单独记录验证方式和结果。
- Wave 可以由多个执行会话并行完成，但每个任务必须单独记录执行位置、base commit、HEAD、diff 范围和验证结果。
- 如果某个任务无法验证，只能影响该任务，不应直接否定或通过整个 Batch / Wave。
- Review Hub 可以汇总验证证据，但结论必须逐任务输出。

## 示例

差：

```text
优化启动流程。
```

好：

```text
运行应用后主窗口可打开；
缺少配置时显示错误提示；
关闭窗口后进程退出；
不影响旧入口。
```

1. 待填写

- 待填写

- 待填写

- 待填写

- 待填写

- 测试命令 / fixture / 日志 / 截图 / 录屏 / CSV / 输出文件 / 用户操作步骤 / 待确认

- 待填写

- 无

## 不适用的写法

如果验证不适用，必须说明原因：

```text
验证类型：不适用
原因：本任务只补充流程模板，不改变项目行为。
替代检查：检查文档链接、字段完整性和模板可复制性。
```
