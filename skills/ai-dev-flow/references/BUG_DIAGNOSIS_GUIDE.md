# Bug 诊断（bug_diagnosis）指南

Bug 诊断（`bug_diagnosis`）用于根因不清、反馈失败、回归疑似或真实环境问题进入修复前的证据优先分析。它不是新任务状态，也不替代审查任务（`review_task`）、修复任务（`repair_task`）或审查-修复循环（`review_repair_loop`）。

## 适用场景

- 用户反馈 bug、慢、不符合预期或验收失败。
- 自动验证失败，但失败原因不清楚。
- 修复前无法判断是否属于当前 TASK 范围。
- 真实环境问题已通过实机测试信号复现（`real_env_signal`）形成 RED / GREEN / SIGNAL。
- 同一问题反复返工，需要先建立可证伪假设。

## 不适用场景

- 已有明确 Review 结论和修复清单，应进入修复任务（`repair_task`）。
- 只是格式、文案或低风险文档修正。
- 用户已经明确要求只记录反馈，不诊断。
- 需要扩大任务范围、修改架构、创建新 TASK 或改变已确认状态，但尚未获得用户确认。

## 诊断阶段

1. 收集现象：记录用户反馈、环境、输入、步骤、期望结果、实际结果和证据位置。
2. 建立反馈信号：定义 RED 失败信号、GREEN 通过信号、SIGNAL 证据来源。
3. 复现与最小化：优先用自动命令、fixture、样例文件 replay 或用户实机 HITL 复测缩小范围。
4. 列出假设：给出 3 到 5 个可证伪假设，每个假设都写证据、验证方式和当前可信度。
5. 插桩验证：必要时添加最小临时诊断日志，必须有唯一前缀，并在修复后清理。
6. 最小修复：只在证据足够且范围明确时进入修复任务（`repair_task`）或审查-修复循环（`review_repair_loop`）。
7. 回归验证：证明 RED 消失、GREEN 成立，并记录验证命令或 HITL 回传证据。
8. 清理临时内容：删除临时日志、调试脚本、临时 fixture 或未确认配置。
9. 复盘记录：写明根因、修复范围、仍缺少的测试缝隙（test seam）和后续任务建议。

## 停止条件

- 无法建立 RED / GREEN / SIGNAL。
- 证据不足，无法判断是否属于当前 TASK 范围。
- 需要用户真实环境、真实账号、真实设备或真实文件才能继续。
- 修复需要扩大范围、改架构、新增依赖或改变已确认状态。
- 第 2 轮后未通过 `CORE.md` progress gate，或第 3 轮后仍未 GREEN。
- 没有测试缝隙，需要转架构巡检（`architecture_review`）或拆出测试缝隙任务。

## 临时诊断日志规则

- 日志前缀必须唯一，例如 `[ADF-DIAG:TASK-123]`。
- 只记录定位所需的最小信息，不记录密钥、Token、账号、真实隐私数据或完整用户文件内容。
- 修复后必须清理临时日志和临时插桩。
- 如果必须保留诊断日志，应转为正式日志并经过任务范围确认。
